## 1. 这个应用是什么

一个**演示型**的安全移动支付终端。一边是模拟的手机 App（左侧 iPhone 框），一边是模拟的银行风控管理后台（右侧），让你能在同一个浏览器页面里同时看到：

- 用户的日常操作（注册、登录、转账、充值、二次验证）
- 这些操作背后调动了哪些密码学算法（ECDH-P256、AES-256-GCM、HKDF-SHA256、ECDSA-P256、HMAC-SHA256、PBKDF2、TOTP）
- 后端如何识别并拦截各种攻击（重放、篡改、撞库、签名伪造、未受信设备…），所有事件以 HMAC 哈希链落盘审计

不是真的银行系统：所有数据在本地 MySQL，账户初始余额 ¥10,000，银行卡充值不会真扣钱。

## 2. 5 分钟快速上手

前置条件：已经按 [`DATABASE.md`](./DATABASE.md) 建好库、按 [`README.md`](./README.md) 起好 Flask（`http://127.0.0.1:5000`）。

```text
1. 浏览器打开 http://127.0.0.1:5000
2. 在手机端"注册"标签填： 用户名 = alice / 登录密码 = Passw0rd! / 支付密码 = pay123 / 手机号 = 13800000001 → 注册
3. 切回"登录"标签，登录 alice
4. 点首页"充值" → 卡号 4242 4242 4242 4242 / MM 12/30 / CVV 123 / 金额 1000 / 支付密码 pay123 → 确认
5. 看右侧管理后台审计日志：能看到 "TOPUP_SUCCESS" 一行，余额变 ¥11,000
```

再注册一个 bob，登录 alice 后用"转账"给 bob 200 元，你会看到双方账单都出现这笔记录（alice 红 -¥200，bob 绿 +¥200），管理后台同时多出两条审计 (`TRANSFER_SUCCESS` + `TRANSFER_INCOMING`)。

## 5. 操作流程详解

### 5.1 注册账户

1. 浏览器打开应用，会自动跳到登录页。点上方"**注册**"切换标签。
2. 填写：

   | 字段 | 要求 | 处理方式 |
   |---|---|---|
   | 用户名 | 唯一 | 明文存 |
   | 登录密码 | ≥ 8 位 | PBKDF2-SHA256 60 万轮 + 16B salt 摘要存 |
   | 支付密码 | ≥ 6 位 | 独立 PBKDF2-SHA256 摘要存(与登录密码不同) |
   | 手机号 | 可选 | **AES-256-GCM 加密落库**，AAD 绑定用户名 |

3. 点 **注册**。后台审计日志会出现 `🟢 [INFO][USER_REGISTER] 用户 xxx 注册成功`。

> 设计点：登录密码和支付密码分开。即便 cookie 被偷、登录态被劫持，转账时还需要支付密码，攻击者依然动不了钱。

---

### 5.2 登录

1. 登录页输入用户名 + 登录密码 → **登录**。
2. 如果你之前**没**绑定 2FA，登录成功，进入首页。审计日志：`🟢 [INFO][LOGIN_SUCCESS] 用户登录`。
3. 如果你绑定过 2FA：返回 `TOTP_REQUIRED`，登录页会多出一栏"动态验证码"，输入 Authenticator 上当前 6 位即可。
4. 登录成功后：
   - 服务器用 **ECDSA-P256 私钥**给你签发一个 **ES256 JWT**（30 分钟有效），存浏览器 localStorage。
   - 你的浏览器**首次**会自动生成一对**本机长期 ECDSA-P256 密钥对**（私钥 / 公钥），存 localStorage；同时把公钥 + 设备指纹（基于 UA + canvas + 屏幕 + 时区 SHA-256）一起 POST 到服务端登记为"受信设备"。
   - 审计日志会多出 `🟢 [INFO][DEVICE_BIND] 绑定设备指纹 xxx…`。

> 失败提示语义：
> - "用户名或密码错误"：基本盘
> - "请输入动态验证码"：触发了 2FA 二阶段
> - "请求过于频繁"：1 分钟内错密码超过 10 次，被 Rate-limit 拦了

---

### 5.3 开启二次验证 (2FA)

1. 登录后进入 **首页** → 第 4 格"安全" → 滚到底部点 **绑定 2FA (TOTP)**。
2. 页面显示：
   - 一个 base32 编码的 `Secret`
   - 一个 `otpauth://totp/SecurePay:<你的用户名>?secret=…` URI

3. 打开手机上的 Google Authenticator / Microsoft Authenticator / 1Password / Authy，**手动添加**新条目，把 secret 填进去（或者把 URI 转 QR 码扫一下）。
4. 在 App 上读到当前的 6 位动态码，回到本页填入 **输入 6 位验证码** → **确认绑定**。
5. 看到"2FA 绑定成功"即生效。下次登录、下次转账都会要求输入动态码。

> 服务端把 secret 用 **AES-256-GCM** 加密（AAD = 你的用户名）落库到 `users.totp_ciphertext`。即便整张表被拖走，也解不出 secret，因为 KEK 在文件系统另一处 0600 保护。

**关闭 2FA**：当前 UI 不提供一键关闭。要关，需要进 MySQL `UPDATE users SET totp_enabled=0, totp_ciphertext=NULL, totp_iv=NULL, totp_tag=NULL WHERE id=?`。

---

### 5.4 银行卡充值

1. 首页第 3 格 **充值**（或底部 Tab "充值"）。
2. 表单：

   | 字段 | 要求 | 提示 |
   |---|---|---|
   | 银行卡号 | 13–19 位数字，会自动按 4 位空格分组 | 测试卡：`4242 4242 4242 4242` |
   | 持卡人姓名 | 任意非空 | |
   | 有效期 | `MM/YY`，自动补斜杠 | 未过期即可，如 `12/30` |
   | CVV | 3 或 4 位数字（密码框遮蔽） | 任意，如 `123` |
   | 充值金额 | > 0 且 ≤ 50000 | 单笔上限 ¥50,000 |
   | 支付密码 | 跟注册时一致 | |
   | 动态验证码 | 仅 2FA 启用时必填 | 6 位 |

3. 点 **确认充值**。前端先在本地用 **Luhn 算法** 校验卡号 → 通过才加密上送。
4. 服务端走完整安全管线（ECDH 协商 → AES-GCM 解密 → 再次 Luhn → 有效期 → CVV → 支付密码 → TOTP → 设备指纹 → ECDSA 验签），通过即入账。
5. 成功后：
   - 余额增加，审计 `🟢 [INFO][TOPUP_SUCCESS] 充值成功 TOPxxx: 银行卡 ****4242 → 金额 1000.00`。
   - 首页"最近交易"出现一条 **+¥1000.00**（绿色），收款名为 `银行卡 ****4242`（只显示后四位）。

> **零持久化承诺**：服务端永远不会把你的完整卡号、CVV、有效期写入数据库、日志、JWT、响应任何位置；只把 `****<last4>` 这种掩码写入流水。哪怕拖库，也还原不出任何完整卡号。

---

### 5.5 向其他用户转账

1. **先确认有收款人**：转账要求"收款方必须是本系统已注册用户名"。可以注册一个 `bob` 作为收款方先。
2. 登录 alice，首页第 2 格 **转账**（或底部 Tab "转账"）。
3. 填写：

   | 字段 | 要求 |
   |---|---|
   | 收款方账户/姓名 | bob 的用户名 |
   | 金额 (CNY) | > 0 |
   | 支付密码 | alice 的支付密码 |
   | 动态验证码 | 仅 alice 启用 2FA 时必填 |

4. 点 **确认转账**。前端日志：
   - `1) 获取 Nonce + 服务器 ECDH 公钥`
   - `2) ECDH-P256 协商 + AES-256-GCM 包裹 (payment_pwd + totp)`
   - `3) ECDSA-P256(SHA-256) 客户端签名交易摘要 (抗抵赖)`

5. 后端通过 **7 重校验**（JWT → Rate-limit → Nonce → AES-GCM → 凭证 → 设备 → ECDSA 签名）后做双账本：

   ```
   alice.balance -= amount
   bob.balance   += amount
   插入 alice 那条 transactions: direction=OUTGOING, receiver=bob
   插入 bob   那条 transactions: direction=INCOMING, receiver=alice
   ```

6. 双方各自的"首页最近交易"会立即出现这笔记录：
   - alice 视角：`bob   -¥200.00` 灰色
   - bob 视角：`来自 alice  +¥200.00` 绿色

> **常见失败场景**：
>
> | 提示 | 含义 |
> |---|---|
> | `余额不足: 当前 ¥xx, 需要 ¥yy` | 字面意思 |
> | `收款方 xxx 不存在` | 收款方用户名拼错或未注册 |
> | `不能向自己转账` | 收款方 = 发款方 |
> | `支付密码错误` | 支付密码填错 |
> | `动态码错误` | 2FA 6 位数填错或时间窗口过了 |
> | `设备未受信` | 当前浏览器没在这个账户上登记过(换浏览器或清了 localStorage 会触发，重新登录即可解决) |
> | `交易签名校验失败` | 本机签名公钥与服务端登记不一致 |
> | `请求非法，请刷新重试` | 重放检测命中（可能你点了两次） |

### 5.6 一键支付（¥38 demo）

1. 首页第 1 格 **收付款** → 进入固定支付页（向"上海咖啡馆"支付 ¥38）。
2. 点 **立即支付**。这是个"商家收款"演示，**不需要**指定收款用户名（receiver 是固定字符串），也不需要支付密码或 2FA。
3. 安全管线还是完整的：Nonce 防重放 + ECDH + AES-256-GCM 包裹 + 一次扣款。
4. 看到"支付成功"即可，余额 -¥38。

> 注：这个模块是为了展示"基础的加密支付报文"流程，使用门槛低。要看完整安全管线请用 §5.5 转账或 §5.4 充值。

---

### 5.7 查看个人安全中心

1. 首页第 4 格 **安全**（或底部 Tab "我的"）。
2. 看到 4 张卡片：

   | 卡片 | 内容 |
   |---|---|
   | 用户 | 你的用户名 |
   | 2FA | "已启用" / "未启用" |
   | 设备指纹 (SHA-256) | 64 位 hex，浏览器熵源算出来的 ID |
   | 本机签名公钥 (ECDSA-P256/SPKI) | 你的浏览器长期密钥对的公钥 PEM |

3. 下面是两个按钮：
   - **绑定 2FA**（或"重新绑定 2FA"，根据当前状态）
   - **退出登录**

> "设备指纹"和"签名公钥"在每次转账都会被比对。你换浏览器、隐身模式、清了 localStorage 后这两个值都会变 — 服务端就会拦截后续转账（因为不再是受信设备）。重新登录会触发"再次登记"流程，过了就行。

---

### 5.8 退出登录 / 切换账户

1. 安全中心底部 → **退出登录**。
2. 本地 JWT 立刻清除，回到登录页。审计 `🟢 [INFO][LOGOUT] 用户登出`。
3. 想切换账户直接用新用户名登录即可。

> 切换账户**不会**清除浏览器的长期 ECDSA 密钥对 — 它绑定的是"这台浏览器"。如果你想模拟"另一台设备登录同一个账户"，请用隐身窗口或换浏览器，这会触发新设备登记。

## 6. 管理后台与审计日志

右半屏。从上到下：

### 6.1 顶栏

- `风险警报: N` — 最近 50 条审计日志里 CRITICAL 等级的条数
- `核心私钥: 安全` — 静态标识，提醒你服务端密钥是在 `secure_storage/` 0600 保护下的

### 6.2 三块统计

- **今日交易额** — 今天 `direction=OUTGOING`（含转账发出、充值不算"对外"）的成功金额合计
- **活跃用户** — 在 `transactions` 表里出现过的不同 `user_id` 数
- **防重放拦截** — `audit_logs.event_type='REPLAY_ATTACK'` 的累计条数

### 6.3 报文加密 / 身份与抗抵赖

两张算法标签卡，给评审者一眼看到系统用了哪些密码学原语。

### 6.4 实时安全审计日志

最近 50 条审计行，按时间倒序。每行格式：

```
[2026-05-17 10:42:11][EVENT_TYPE] 人读描述
```

颜色：

- 🟢 emerald — `INFO`（正常业务事件）
- 🟡 yellow — `WARN`（业务失败：错密码、Luhn 失败、JWT 过期、Rate-limit 触发）
- 🔴 red bold — `CRITICAL`（高危：重放、解密失败、未受信设备、签名伪造、系统异常）

每条审计写入时都会算 `operation_hash = HMAC-SHA256(prev_hash ‖ event ‖ user_id ‖ desc ‖ ip ‖ ts)`，组成哈希链。任何一行被改后续 hash 全断，可作为内部巡检和取证依据。

## 7. 背后的安全机制（用户视角）

| 你做了什么 | 服务端做了什么 | 为什么这么麻烦 |
|---|---|---|
| 输登录密码点登录 | PBKDF2-SHA256 60 万轮校验 hash | 即便拖库，撞库代价高 |
| 输了 6 位 TOTP | HMAC-SHA1(30s 窗口) 校验 | 即便密码泄露也登不上 |
| 浏览器静默生成签名密钥对 | 把公钥登记到 `users.signing_pub_key` | 后续每笔转账都用私钥签，事后不可抵赖 |
| 转账时填了支付密码 | 独立 PBKDF2 校验 | 跟登录密码区分，session 劫持不致命 |
| 转账时浏览器签了一份交易摘要 | 用登记公钥验签 | 服务端只信"用户本机签的字"，不信"session 状态" |
| 充值填了卡号 + CVV | ECDH+AES-GCM 包裹整包上送 → 服务端校验后丢弃 | 卡号一辈子不会落数据库 |
| 后台拿到请求 | 先看 nonce 是不是用过 | 防止有人录了上次请求再发一遍 |
| 你看到余额变化 | 后端在同一事务里加锁双方账户、写哈希链审计 | 并发安全 + 取证可信 |

## 8. 攻击演示与解读

我们写了一个 `attack_sim.py` 风格的脚本，能跑 8 种典型攻击。下面表格描述每种攻击会在审计日志里留下的痕迹（对应 [攻击演示](#8-攻击演示与解读) 这次专题）：

| 攻击类型 | 后端怎么发现的 | 审计日志看到什么 |
|---|---|---|
| **撞库** (5 次错密码) | PBKDF2 hash 不匹配 | `🟡 LOGIN_FAIL × N` |
| **重放** (同 nonce 二次提交) | `used_nonces FOR UPDATE` 命中 used_at 非空 | `🔴 REPLAY_ATTACK 拦截重放: Nxxx`，统计区"防重放"+1 |
| **报文篡改** (改 ciphertext 1 字节) | AES-GCM tag 校验失败 | `🔴 PAY_FAIL 解密失败: MAC check failed` |
| **错支付密码** | PBKDF2 不匹配 | `🟡 TRANSFER_FAIL 支付密码错误` |
| **未受信设备** | `user_devices` 查不到 (user_id, fp, trusted=1) | `🔴 TRANSFER_FAIL 未受信设备 xxx…` |
| **错 ECDSA 签名** | 服务端用登记公钥验签失败 | `🔴 SIGN_FAIL 客户端签名校验失败` |
| **JWT 伪造** | ES256 验签失败 | `🟡 AUTH_FAIL JWT 校验失败: Signature verification failed` |
| **假银行卡** | Luhn 算法不通过 | `🟡 TOPUP_FAIL 卡号 Luhn 校验未通过` |

风险警报数 = CRITICAL 行数。在 8 次攻击中，**重放 / 报文篡改 / 未受信设备 / 签名伪造** 这 4 种是 CRITICAL（明显是恶意），其它 4 种归为 WARN（可能用户手滑也可能恶意，需结合频次判断）。

## 9. 故障排查与 FAQ

**Q: 我打开浏览器，看到的不是登录页，是 demo_user 的首页（带余额 ¥9998 之类的）？**
A: 这是 `/api/dashboard` 在没 JWT 时回落到 demo_user 的设计（防止首屏空荡）。直接点"安全 → 退出登录"或按下方"登录"按钮即可。

**Q: 转账提示"设备未受信"，但我刚才明明能登录？**
A: 你换了浏览器 / 隐身模式 / 清了 localStorage。当前账户在新浏览器的指纹没登记。退出登录再登一次，会触发自动登记。

**Q: 2FA 验证码总是错？**
A: 三个可能：
1. 手机和服务器时钟差 > 30 秒 — 同步一下时间。
2. 你看的是"旧"那 30 秒窗口的码 — 等下一个码出现再填。
3. 你扫的 QR 来自别的账户的 setup — 重新点"重新绑定 2FA"再来一遍。

**Q: 转账失败提示"请求过于频繁"？**
A: 触发了 Rate-limit。转账接口默认 10 次 / 分钟，登录 10 次 / 分钟，充值 5 次 / 分钟。等一分钟就好。

**Q: 我注册时填的手机号，能在哪里看回？**
A: 当前 UI 没有"查看手机号"的入口。后端是用 KEK 加密落库的（`users.phone_ciphertext`），需要时直接调 `cu.decrypt_field`，不会通过 HTTP 接口暴露。

**Q: 重启 Flask 后我的 JWT 还有效吗？**
A: **有效**。JWT 是用 ECDSA 私钥签发的，重启后私钥还在 `secure_storage/jwt_ecdsa_private.pem`，所以 token 直到 30 分钟过期前都能用。但 Flask session（cookie）会失效，因为 `app.secret_key` 取自环境变量 `FLASK_SECRET`，没设置时每次启动都换。

**Q: `secure_storage/` 里那堆文件能删吗？**
A: 不建议。删了之后：
- `kek_v1.bin` 删 → 已加密的 `users.phone_ciphertext` / `totp_ciphertext` 永远解不开
- `jwt_ecdsa_private.pem` 删 → 旧 JWT 全部失效，下次启动会生成新私钥
- `audit_hmac.key` 删 → 审计哈希链断代，之前的 hash 没法继续验

完整重建参见 [`DATABASE.md` §6 重置](./DATABASE.md)。

**Q: 我想看哈希链是怎么续上的？**
A: 进 MySQL 跑：
```sql
SELECT id, severity, event_type,
       SUBSTRING(operation_hash, 1, 12) AS hash,
       SUBSTRING(prev_hash, 1, 12)      AS prev
FROM audit_logs ORDER BY id DESC LIMIT 10;
```
每行的 `prev` 就是上一行的 `hash` 前 12 位。
